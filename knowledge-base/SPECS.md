# BOS Genesis MoP Execution Agent - SPECS

**Document status:** Draft v0.1  
**Document name:** `SPECS.md`  
**Agent name:** `bosgenesis-mop-execution-agent`  
**Alternative name:** `bosgenesis-k8s-mop-installer-agent`  
**Python package:** `bosgenesis_mop_execution_agent`  
**Primary mode:** Async, long-running, externally controlled execution worker  
**Reasoning posture:** No autonomous reasoning authority  
**Execution posture:** Deterministic worker with strict mechanical guardrails  
**Primary input:** MoP Creation Agent output bundle  
**Primary output:** Audited Kubernetes/Helm installation execution, validation report, rollback report, execution notes, and release notes  
**Target platform:** Kubernetes namespace recreation through governed MCP servers  

---

## 1. Intent

This specification defines the behavior, boundaries, contracts, APIs, memory model, safety posture, and acceptance criteria for the `bosgenesis-mop-execution-agent`.

The agent is the downstream executor for artifacts generated by `bosgenesis-mop-creation-agent`. The MoP Creation Agent reverse engineers a Kubernetes namespace and produces a human Method of Procedure, professional PDF, Markdown installation notes, generated manifests, Helm values, and a standalone `machine_execution_plan.yaml`. This execution agent consumes those artifacts and installs or recreates the Kubernetes and Helm resources into a target namespace.

The execution agent is intentionally not a reasoning agent. It is a deterministic, policy-enforcing worker. The external LLM or orchestration layer is the brain. The execution agent is the hands, state machine, safety boundary, memory system, and audit trail.

```text
The agent may be autonomous in execution, but not autonomous in reasoning.
```

```text
Memory is allowed to remember.
Memory is not allowed to decide.
```

## 2. Product Context

The BOS Genesis platform includes multiple MCP servers and agents:

| Component | Role |
|---|---|
| `bosgenesis-helm-manager-mcp` | Manages Helm repositories, chart rendering, dry-runs, installs, upgrades, status, history, rollback, and uninstall operations. |
| `bosgenesis-k8s-data-ingestion-agent` | Scans Kubernetes namespaces on a schedule and ingests observed state into PostgreSQL and ClickHouse. |
| `bosgenesis-k8s-inspector-agent` | Inspects and manages Kubernetes namespace-scoped resources. |
| `bosgenesis-mop-creation-agent` | Reverse engineers Kubernetes/Helm namespace state and creates MoP artifacts. |
| `bosgenesis-release-note-agent` | Generates release notes, execution summaries, and operational change notes. |
| `bosgenesis-mop-execution-agent` | New worker described by this specification. It executes MoP installation plans under external LLM control. |

The MoP Creation Agent is explicitly a generator, not an executor. This execution agent closes that loop by installing the generated artifacts, while preserving human approval, namespace scope, dry-run-before-mutation, no Secret value copying, no production data copying, deterministic guardrails, and full auditability.

## 3. Terminology

| Term | Meaning |
|---|---|
| MoP | Method of Procedure generated by `bosgenesis-mop-creation-agent`. |
| Bundle | A MoP output directory, uploaded archive, or artifact reference containing `machine_execution_plan.yaml`, Markdown/PDF MoP artifacts, generated YAML files, metadata, and optional Helm values. |
| Machine Execution Plan | Canonical YAML contract used by this agent to determine phases, steps, dependency order, command references, expected outcomes, validation, and rollback metadata. |
| External LLM Controller | The caller or orchestration brain that reasons about failures and sends explicit instructions to this worker. |
| Human Approver | A human who authorizes mutating, destructive, rollback, or policy-exception actions. |
| Worker | This execution agent. It performs deterministic parsing, policy checks, MCP calls, state transitions, memory writes, observations, and audit logging. |
| Observation | Structured facts emitted after each check, dry-run, MCP call, mutation, validation, wait, timeout, or error. |
| Decision Required | A job state where the worker has paused and requires an external LLM instruction, human approval, or both. |
| Guardrail | Deterministic rule enforced by the worker regardless of external LLM instructions. |
| Execution Memory | Memory used for continuity, auditability, recall, and external-LLM context. It has no decision authority. |

## 4. Non-Negotiable Design Principles

### 4.1 No autonomous reasoning authority

The worker must not independently decide how to repair, reinterpret, skip, retry, reorder, patch, roll back, or ignore a failed step.

Allowed worker decisions are mechanical only:

- schema valid or invalid;
- namespace scope allowed or blocked;
- approval present or missing;
- dry-run required or satisfied;
- command matches instruction or does not match instruction;
- Secret redaction required;
- concurrency lock available or unavailable;
- timeout exceeded;
- audit event written or failed;
- MCP response success or failure;
- job state transition permitted or denied.

Forbidden worker decisions include:

- how to fix a failed manifest;
- whether an unexpected error is safe to ignore;
- whether to change YAML semantics;
- whether to delete and recreate a resource;
- whether to override immutable field conflicts;
- whether to proceed after unexpected cluster state;
- whether to infer a Helm chart, version, values file, or repository;
- whether to alter dependency order for semantic reasons;
- whether to execute a rollback without explicit instruction and approval.

### 4.2 Brainless does not mean unsafe

```text
The worker must not infer solutions, but it must enforce schema, scope,
approval, idempotency, redaction, concurrency locks, timeout limits,
and audit logging.
```

### 4.3 Machine plan first

The worker must parse `machine_execution_plan.yaml` before using any other artifact. Human MoP Markdown and installation notes are supporting context. YAML manifest files and Helm values are the source of resource content. The professional PDF is a human review artifact only.

### 4.4 Dry-run before mutation

Every mutating Kubernetes or Helm action must have a successful dry-run or equivalent preflight result recorded in the current job attempt before mutation can be executed.

### 4.5 Human approval before mutation

No mutating action may be executed unless a valid human approval record exists for that exact job, phase, step, namespace, action type, and approval scope.

### 4.6 Target namespace only

All resource creation, update, delete, validation, and logs/events collection must stay inside the target namespace unless an explicitly approved future version enables cluster-scoped or multi-namespace scope.

### 4.7 No Secret values and no production data

The worker must never copy Kubernetes Secret values, database rows, MongoDB documents, Kafka messages, Redis values, object-store files, persistent-volume contents, or business payloads. Secret placeholders may be created only when the plan and human approval permit it, and values must come from approved secure sources outside this agent.

### 4.8 Memory cannot decide

All memory layers are allowed for continuity, recall, audit, and external-LLM context. Memory results must be packaged as evidence for the external LLM. The worker must not turn memory into direct execution decisions.

## 5. Scope

### 5.1 In scope for v1

- Create async execution jobs from MoP output bundles.
- Accept bundle references from local storage, uploaded zip, artifact manifest, or MoP Creation Agent API/MCP response.
- Parse `machine_execution_plan.yaml` as the canonical execution plan.
- Parse human MoP Markdown and machine-readable installation notes as supporting context.
- Read generated Kubernetes YAML resources.
- Read generated Helm values files where present.
- Read `artifact.json`, `artifact-index.json`, and `response.json` for evidence, metadata, warnings, and trace linkage.
- Validate artifact completeness and checksum metadata where available.
- Enforce target namespace only.
- Enforce dry-run before mutation.
- Enforce human approval before mutation.
- Execute Kubernetes operations through `bosgenesis-k8s-inspector-agent`.
- Execute Helm operations through `bosgenesis-helm-manager-mcp`.
- Query recent and historical namespace facts through `bosgenesis-k8s-data-ingestion-agent`.
- Generate execution summaries and release notes through `bosgenesis-release-note-agent`.
- Maintain async long-running jobs.
- Support pause, resume, cancel, retry-by-instruction, continue-by-instruction, and rollback-by-instruction.
- Emit structured observations after every operation.
- Pause in `decision_required` state when reasoning is needed.
- Provide observation context to the external LLM.
- Persist job state durably.
- Persist execution memory without secrets.
- Persist immutable audit events.
- Expose REST and MCP interfaces.
- Emit logs, metrics, traces, and spans.
- Support namespace-scoped concurrency locks.
- Support idempotency keys for job creation and mutation requests.
- Produce final execution report, validation report, rollback report if used, and release/execution notes.

### 5.2 Out of scope for v1

- Autonomous reasoning or remediation by the worker.
- Direct use of `kubectl` or `helm` outside governed MCPs unless explicitly enabled for local development only.
- Cluster-scoped resource creation or modification.
- Multi-namespace execution.
- Secret value migration.
- Production data migration or PVC data copying.
- Database schema/data migration.
- Automatic application-level data seeding.
- Private Helm repository inference.
- Docker image rebuilding.
- Automatic Qdrant ingestion of execution memory without approval.
- Self-directed rollback without explicit instruction and approval.
- Human approval simulation.
- Ignoring, suppressing, or auto-resolving policy violations.

## 6. Actors and Authority Model

| Actor | Authority | Cannot Do |
|---|---|---|
| External LLM Controller | Decide next step, remediation strategy, retry strategy, wait strategy, patch strategy, skip/continue strategy, rollback strategy, escalation summary. | Bypass guardrails, bypass human approval, execute outside scope, access Secret values through this worker. |
| Human Approver | Authorize mutating actions, destructive actions, namespace creation, rollback, PVC deletion, policy exceptions. | Override hard-coded forbidden operations unless product policy allows an explicit exception type. |
| Worker Agent | Parse, validate, execute instructed operations, enforce policy, call MCPs, emit observations, persist memory, audit. | Reason, infer, repair, skip, continue after unexpected failure, approve changes. |
| MCP Servers | Perform governed Kubernetes/Helm/data/release-note operations. | Override this agent's policy state or approval model. |
| Operator/API Client | Submit jobs, upload bundles, inspect state, provide approvals and LLM instructions. | Force execution without valid state transition and policy checks. |

## 8. Artifact Input Contract

### 8.1 Required bundle files

| File | Required | Authority | Usage |
|---|---:|---|---|
| `machine_execution_plan.yaml` | Yes | Canonical execution contract | Parsed first; defines phases, steps, dependency graph, command metadata, expected outcomes, validation, rollback, and executor constraints. |
| Human-readable MoP Markdown | Yes | Supporting context | Used for human sequencing context, checklist, validation wording, rollback notes, and operator-facing descriptions. |
| Generated Kubernetes YAML files | Conditional | Resource content | Applied or dry-run through governed Kubernetes MCP when referenced by the machine plan. |
| Helm values files | Conditional | Helm input | Used by Helm MCP when Helm phases exist. |
| `artifact.json` | Recommended | Evidence metadata | Used for classification, warnings, trace IDs, counts, and provenance. |
| `artifact-index.json` | Recommended | File index | Used to locate generated files and validate completeness. |
| `response.json` | Recommended | Run metadata | Used for generation status, source/target namespace, paths, memory status, warning counts. |
| Machine-readable installation notes Markdown | Recommended | Supporting context | Used as fallback if standalone YAML is missing, and as additional context. |
| Professional MoP PDF | Optional | Human review only | Must not be parsed as canonical execution input. |

### 8.2 Bundle validation rules

The worker must validate:

- bundle is readable;
- required files exist;
- machine plan schema version is supported;
- target namespace in job request matches machine plan unless a safe namespace rewrite policy is explicitly enabled;
- generated YAML file references exist;
- resource kinds are allowed for v1;
- every manifest is namespace-scoped;
- every namespace field is the target namespace or safely rewritten to the target namespace by deterministic normalization;
- no Secret values are present;
- no suspicious base64-like Secret payloads are present in manifests, logs, memory, or reports;
- no cluster-scoped resources appear in the executable set;
- no production data migration commands appear;
- plan phases form an acyclic dependency graph;
- mutating steps declare dry-run/preflight commands;
- mutating steps declare required human approval policy;
- rollback steps are present or marked unavailable with explanation.

### 8.3 Artifact authority order

```yaml
authority_order:
  - explicit_human_approval
  - explicit_external_llm_instruction
  - machine_execution_plan_yaml
  - generated_yaml_and_values_files
  - artifact_json_evidence
  - machine_readable_installation_notes
  - human_mop_markdown
  - historical_execution_memory
  - prior_qdrant_or_vector_references
```

The authority order does not allow unsafe execution. Guardrails always override all inputs.

## 9. Job Lifecycle

### 9.1 Job states

```yaml
job_states:
  - created
  - bundle_validating
  - bundle_invalid
  - awaiting_human_approval
  - awaiting_external_llm_instruction
  - ready
  - running
  - waiting
  - decision_required
  - paused
  - cancelling
  - cancelled
  - rollback_requested
  - rollback_running
  - rollback_decision_required
  - rollback_completed
  - completed
  - completed_with_warnings
  - failed
```

### 9.2 State transition rules

- `created` may transition only to `bundle_validating` or `cancelled`.
- `bundle_validating` may transition to `bundle_invalid`, `awaiting_human_approval`, `awaiting_external_llm_instruction`, or `ready`.
- `ready` may transition to `running`, `paused`, or `cancelled`.
- `running` may transition to `waiting`, `decision_required`, `paused`, `completed`, `completed_with_warnings`, `failed`, or `rollback_requested`.
- `decision_required` may transition only after an external LLM instruction, human approval, or cancellation.
- Any mutating transition must verify approval, dry-run, lock, target namespace, and policy.
- `rollback_running` must follow the same guardrails as forward execution.

### 9.3 Step states

```yaml
step_states:
  - pending
  - blocked
  - ready
  - dry_run_running
  - dry_run_succeeded
  - dry_run_failed
  - approval_required
  - approved
  - mutation_running
  - mutation_succeeded
  - mutation_failed
  - validation_running
  - validation_succeeded
  - validation_failed
  - waiting
  - decision_required
  - skipped_by_instruction
  - cancelled
```

## 10. External LLM Instruction Contract

The external LLM must send explicit, bounded instructions. The worker must validate instructions mechanically.

### 10.1 Instruction envelope

```yaml
instruction:
  schema_version: "1.0"
  job_id: "uuid"
  instruction_id: "uuid"
  correlation_id: "string"
  issued_by: "external_llm_controller"
  issued_at: "RFC3339"
  target_namespace: "string"
  phase_id: "string"
  step_id: "string"
  intent: "retry|continue|skip|patch_manifest|wait|rollback|collect_more_observations|abort|escalate"
  rationale_summary: "short non-secret explanation from external LLM"
  allowed_actions:
    - action_type: "dry_run|apply|helm_install|helm_upgrade|helm_rollback|delete|validate|wait|inspect"
      resource_ref: "optional resource reference"
      command_fingerprint: "optional sha256"
  patch:
    format: "json_patch|strategic_merge|yaml_replacement|none"
    content_ref: "artifact or inline redacted patch"
  wait_policy:
    duration_seconds: 300
    poll_interval_seconds: 15
  safety_expectations:
    target_namespace_only: true
    dry_run_before_mutation: true
    human_approval_before_mutation: true
    no_secret_values: true
  expires_at: "RFC3339"
```

### 10.2 Instruction validation

The worker must reject or pause if:

- schema is invalid;
- instruction refers to a different job;
- target namespace does not match job target namespace;
- phase or step is unknown;
- instruction is expired;
- requested action violates policy;
- requested mutation lacks matching human approval;
- requested mutation lacks successful dry-run;
- command fingerprint does not match approved command scope;
- patch contains Secret values;
- patch changes resource namespace outside target namespace;
- patch attempts cluster-scoped resources;
- patch changes semantics without approval scope that explicitly includes patching.

### 10.3 Decision-required context envelope

When the worker pauses, it must package facts for the external LLM.

```yaml
decision_required_context:
  schema_version: "1.0"
  job_id: "uuid"
  phase_id: "string"
  step_id: "string"
  reason_code: "yaml_syntax_error|dry_run_failed|resource_exists|immutable_field_conflict|pod_unschedulable|mcp_unavailable|timeout|policy_blocked|validation_failed|unknown"
  target_namespace: "string"
  plan_excerpt:
    phase_objective: "string"
    step_title: "string"
    expected_outcomes: []
  attempted_action:
    action_type: "string"
    command_summary: "redacted string"
    command_fingerprint: "sha256"
  observations:
    - observation_id: "uuid"
      timestamp: "RFC3339"
      source: "k8s_mcp|helm_mcp|data_ingestion_mcp|worker_policy|worker_parser"
      status: "success|failed|warning"
      redacted_stdout: "string"
      redacted_stderr: "string"
      structured_error: {}
  resource_state:
    observed_objects: []
    events: []
    pod_summaries: []
  memory_context:
    similar_past_failures: []
    prior_resolutions: []
    authority: "historical_context_only_not_decision"
  policy_context:
    allowed_next_action_types: []
    blocked_action_types: []
    required_approval_scopes: []
```

    target_namespace_confirmed: true
    no_secret_values_confirmed: true

### 11.2 Approval rules

- Approval must be explicit.
- Approval must be scoped.
- Approval must expire.
- Approval must be bound to job ID and target namespace.
- Approval must be recorded before mutation.

- Approval may not authorize production data copying in v1.
- Approval for delete, rollback, or PVC actions must use a stricter destructive-action scope.

## 13. Memory Model

All layers of memory exist as execution memory. No memory layer has decision authority.

```yaml
memory_policy:
  principle: "memory_can_remember_but_cannot_decide"
  store_secret_values: false
  store_production_data: false
  store_raw_unredacted_logs: false
  require_redaction: true
  require_job_correlation: true
```

| Memory layer | Purpose | Durable | Decision authority |
|---|---|---:|---:|
| In-run state memory | Track current phase, step, attempt, observations, local locks, and pending approvals. | No | No |
| Durable job memory | Resume async jobs after restart. | Yes | No |
| Resource state memory | Store observed K8s/Helm facts before/during/after actions. | Yes | No |
| Episodic execution memory | Store prior non-secret execution outcomes and remediation summaries. | Yes | No |
| Semantic failure memory | Retrieve similar failures and prior external-LLM guidance for context. | Yes | No |
| Policy memory | Store deterministic guardrails and configured policy. | Yes | Guardrail only |
| Approval memory | Record human approval and rejection events. | Yes | Authorization only |
| Audit memory | Immutable event log. | Yes | No |
| Observability memory | Trace IDs, metrics, spans, durations, retry counts, failure taxonomy. | Yes | No |

### 13.1 Memory retrieval rules

- Retrieve memory only for the same tenant/environment and safe context.
- Prefer namespace, application, chart, resource kind, error code, and MCP source filters.
- Do not retrieve raw unredacted output.
- Label memory as `historical_context_only_not_decision`.
- Include retrieved memory in external LLM context only.
- Do not execute an action based solely on memory.

## 14. Policy and Guardrails

### 14.1 Hard blocks

The worker must block:

- cluster-scoped create/update/delete in v1;
- access to Secret `.data` or `.stringData` values;
- production data copy commands;
- PVC data copying;
- namespace mismatch;
- resource namespace outside target namespace;
- commands containing unsafe shell composition when not explicitly modeled;
- direct network calls not covered by configured MCP clients;
- mutation without dry-run;
- mutation without approval;
- expired approval;
- ambiguous instruction;
- direct manual override from external LLM that conflicts with policy;
- audit write failure before mutation.

### 14.2 Soft pauses

The worker must enter `decision_required` when encountering:

- YAML parse error;
- schema validation failure;
- server-side dry-run failure;
- resource already exists with conflicting ownership;
- immutable field conflict;
- Helm chart render failure;
- Helm release exists with different metadata;
- PVC pending;
- pod unschedulable;
- node unavailable;
- CrashLoopBackOff;
- ingress host/path conflict;
- missing secret placeholder;
- timeout waiting for readiness;
- validation mismatch;
- MCP server error;
- unknown warning not explicitly allowed by policy.

### 14.3 Idempotency

- Job creation must support idempotency keys.
- Mutating requests must support idempotency keys.
- The worker must record action fingerprints.
- Replayed requests with the same idempotency key must return the same result or safe duplicate status.
- The worker must not re-run a mutation after an uncertain outcome without external instruction.

## 18. Execution Algorithm

```text
create_job(request):
    validate request schema
    create job record in created state
    acquire idempotency record
    enqueue bundle validation task
    return job_id

worker_loop(job):
    load job state
    acquire target namespace lock when execution begins
    validate bundle
    parse machine_execution_plan.yaml
    load supporting MoP markdown and installation notes
    load manifests and values
    validate policy and namespace scope
    build phase/step graph

    if dry_run_only:
        execute preflight and dry-runs only
        emit observations
        complete or pause for decision_required

    for each ready phase and step:
        if step requires external instruction and none exists:
            transition to awaiting_external_llm_instruction
            emit decision context
            stop

        if step is mutating:
            ensure dry-run succeeded for current attempt
            ensure approval exists and matches scope
            ensure namespace lock is held
            ensure audit pre-event is written

        execute through the correct MCP client
        capture observation
        redact output
        persist observation and audit event

        if MCP returns error, warning, timeout, or ambiguous state:
            transition to decision_required
            package context for external LLM
            stop

        run validation when defined
        if validation fails:
            transition to decision_required
            package context
            stop

    produce final reports
    release lock
    complete job
```

## 19. Error Taxonomy

| Code | Meaning | Default worker action |
|---|---|---|
| `BUNDLE_MISSING_REQUIRED_FILE` | Required artifact missing. | Stop: `bundle_invalid`. |
| `PLAN_SCHEMA_INVALID` | Machine plan failed schema validation. | Stop: `bundle_invalid`. |
| `PLAN_GRAPH_CYCLE` | Phase/step dependency cycle. | Stop: `bundle_invalid`. |
| `NAMESPACE_MISMATCH` | Plan/resource namespace differs from target namespace. | Policy block. |
| `CLUSTER_SCOPE_BLOCKED` | Cluster-scoped resource detected. | Policy block. |
| `SECRET_VALUE_DETECTED` | Secret value or suspicious payload detected. | Policy block. |
| `PRODUCTION_DATA_COPY_DETECTED` | Data copy command detected. | Policy block. |
| `DRY_RUN_FAILED` | Dry-run/preflight failed. | `decision_required`. |
| `APPROVAL_REQUIRED` | Mutating action lacks approval. | `awaiting_human_approval`. |
| `APPROVAL_SCOPE_MISMATCH` | Approval does not match action. | `decision_required`. |
| `RESOURCE_ALREADY_EXISTS` | Resource exists unexpectedly. | `decision_required`. |
| `IMMUTABLE_FIELD_CONFLICT` | Apply failed due to immutable field. | `decision_required`. |
| `HELM_RENDER_FAILED` | Helm render/dry-run failed. | `decision_required`. |
| `POD_UNSCHEDULABLE` | Pod cannot schedule. | `decision_required`. |
| `NODE_UNAVAILABLE` | Node issue detected. | `decision_required`. |
| `PVC_PENDING` | PVC remains pending. | `decision_required`. |
| `INGRESS_CONFLICT` | Ingress host/path conflict. | `decision_required`. |
| `VALIDATION_FAILED` | Post-action validation failed. | `decision_required`. |
| `MCP_UNAVAILABLE` | Required MCP server unavailable. | `waiting` or `decision_required` after timeout. |
| `TIMEOUT_EXCEEDED` | Step/job timeout exceeded. | `decision_required`. |
| `AUDIT_WRITE_FAILED` | Audit event could not be persisted. | Fail closed before mutation. |
| `UNKNOWN_ERROR` | Unclassified error. | `decision_required`. |

## 26. Definition of Done

The first production-ready release is done when:

- all MUST requirements have passing tests;
- sample MoP bundle can run in dry-run-only mode;
- sample MoP bundle can execute in a disposable target namespace with explicit approval;
- injected failures produce `decision_required` and context envelopes;
- no worker remediation occurs without external instruction;
- redaction tests prove no Secret values persist;
- audit completeness tests pass;
- restart/resume tests pass;
- MCP contract tests pass;
- final execution and validation reports are generated;
- deployment and rollback runbooks are available;
- logs, metrics, and traces are visible in configured observability backends.
